1. Resumen Ejecutivo

Este proyecto implementa una aplicación e-commerce completa con arquitectura de microservicios, desarrollada en Spring Boot (Backend) y React (Frontend), simulando la funcionalidad de Mercado Libre. El sistema incluye gestión de productos, reseñas, vendedores, imágenes y un panel de compras interactivo.

Características Principales:
  • Arquitectura RESTful con separación clara de responsabilidades
  • Base de datos SQLite con JPA/Hibernate
  • Frontend responsive con Tailwind CSS
  • Documentación automática con Swagger/OpenAPI
  • Sistema de reseñas y calificaciones
  • Galería de imágenes dinámica
  • Panel de compras estilo Mercado Libre

3.3 Modelo de Datos

Entidad Product

@Entity
@Table(name = "product")
@NamedEntityGraph(
    name = "product-with-details",
    attributeNodes = {
        @NamedAttributeNode("seller"),
        @NamedAttributeNode("reviews")
    }
)
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String name;
    private String description;
    private Double price;
    
    @ElementCollection
    @CollectionTable(name = "product_images", 
                    joinColumns = @JoinColumn(name = "product_id"))
    @Column(name = "image_url")
    private List<String> imageUrls;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "seller_id", nullable = false)
    private Seller seller;
    
    @OneToMany(mappedBy = "product", cascade = CascadeType.ALL)
    private List<Review> reviews;
}

Relaciones de Base de Datos

  • Product ↔ Seller: Many-to-One
  • Product ↔ Review: One-to-Many
  • Product ↔ ProductImages: One-to-Many (ElementCollection)

5. Comunicación Backend-Frontend

5.1 Flujo de Datos

  1. Inicialización: Frontend carga en puerto 5173
  2. Navegación: Usuario accede a /product/{id}
  3. Fetch Request: Frontend hace petición a http://localhost:8080/api/products/{id}
  4. CORS: Backend valida origen y permite petición
  5. Procesamiento: Backend consulta base de datos con EntityGraph
  6. Respuesta: JSON con ProductDetailDTO
  7. Renderizado: Frontend actualiza UI con datos recibidos

5.2 Ejemplo de Comunicación

Frontend - ProductPage.jsx

const fetchProduct = async (id) => {
  try {
    setLoading(true);
    const response = await fetch(`http://localhost:8080/api/products/${id}`);
    
    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }
    
    const data = await response.json();
    setProduct(data);
  } catch (error) {
    setError(error.message);
  } finally {
    setLoading(false);
  }
};

Backend - ProductController.java

@GetMapping("/{id}")
public ResponseEntity<ProductDetailDTO> getProductById(@PathVariable Long id) {
    Optional<Product> product = productRepository.findProductById(id);
    return product.map(p -> ResponseEntity.ok(new ProductDetailDTO(p)))
            .orElse(ResponseEntity.notFound().build());
}

11. Guía para Desarrolladores Junior

11.1 Conceptos Básicos Explicados

¿Qué es una API REST?

Una API REST es como un "menú" que permite a diferentes aplicaciones comunicarse entre sí. En nuestro proyecto:

  • Frontend (React) "pide" información al Backend (Spring Boot)
  • El Backend "sirve" los datos desde la Base de Datos (SQLite)
  • Es como un camarero que toma tu pedido y te trae la comida

¿Qué es CORS?

CORS es una regla de seguridad que permite que el navegador web se comunique con servidores en diferentes dominios. En nuestro caso:

  • Frontend corre en localhost:5173
  • Backend corre en localhost:8080
  • CORS permite que se comuniquen de forma segura

¿Qué es JPA/Hibernate?

JPA es como un "traductor" que convierte objetos Java en datos de base de datos:

  • Entidad: Clase Java que representa una tabla
  • Repositorio: Clase que maneja las operaciones de base de datos
  • DTO: Objeto que se envía al frontend (versión "limpia" de la entidad)

12. Decisiones de Arquitectura

12.1 ¿Por qué Spring Boot?

  • Productividad: Configuración automática y convenciones
  • Ecosistema: Gran cantidad de librerías y documentación
  • Escalabilidad: Fácil de escalar horizontalmente
  • Enterprise: Ampliamente usado en empresas grandes

12.2 ¿Por qué React?

  • Componentes Reutilizables: Código más limpio y mantenible
  • Virtual DOM: Rendimiento optimizado
  • Ecosistema: Gran cantidad de librerías disponibles
  • Curva de Aprendizaje: Relativamente fácil de aprender

12.3 ¿Por qué SQLite?

  • Simplicidad: No requiere servidor separado
  • Desarrollo: Perfecto para prototipos y desarrollo
  • Portabilidad: Un solo archivo contiene toda la base de datos
  • Limitaciones: No recomendado para producción con alta concurrencia

12.4 Alternativas Consideradas

Tecnología Ventajas Desventajas ¿Por qué no elegimos?
Node.js + Express JavaScript en todo el stack Menos maduro para aplicaciones empresariales Spring Boot más robusto para APIs complejas
Angular Framework completo Curva de aprendizaje más alta React más flexible y popular
PostgreSQL Mejor para producción Configuración más compleja SQLite más simple para desarrollo

13. Patrones de Diseño Implementados

13.1 DTO Pattern (Data Transfer Object)

// ❌ Mal: Exponer entidad directamente
@Entity
public class Product {
    private String name;
    private String internalField; // Campo interno que no debe exponerse
}

// ✅ Bien: Usar DTO
public class ProductDetailDTO {
    private String name;
    // Solo campos que necesitamos enviar al frontend
}

Ventajas del DTO Pattern:

  • Seguridad: No exponemos campos internos
  • Flexibilidad: Podemos cambiar la estructura sin afectar la entidad
  • Rendimiento: Solo enviamos los datos necesarios
  • Mantenibilidad: Separación clara de responsabilidades

13.2 Repository Pattern

// Abstracción del acceso a datos
public interface ProductRepository extends JpaRepository<Product, Long> {
    Optional<Product> findProductById(Long id);
}

// Uso en el controlador
@Autowired
private ProductRepository productRepository;

13.3 Entity Graph Pattern

@NamedEntityGraph(
    name = "product-with-details",
    attributeNodes = {
        @NamedAttributeNode("seller"),
        @NamedAttributeNode("reviews")
    }
)

¿Por qué usar Entity Graph?

  • Evita N+1 Queries: Carga todas las relaciones en una sola consulta
  • Mejor Rendimiento: Menos viajes a la base de datos
  • Control: Especificamos exactamente qué cargar

14. Consideraciones de Seguridad

14.1 Vulnerabilidades Comunes

⚠️ IMPORTANTE - Solo para Desarrollo
Este proyecto NO implementa medidas de seguridad para producción.

Vulnerabilidades Identificadas:

  • Sin Autenticación: Cualquiera puede acceder a la API
  • Sin Autorización: No hay control de acceso
  • Sin Validación de Entrada: No validamos datos de entrada
  • Sin HTTPS: Comunicación no encriptada
  • SQL Injection: Aunque JPA ayuda, no es suficiente

14.2 Medidas de Seguridad para Producción

Medida Implementación Beneficio
Autenticación Spring Security + JWT Control de acceso
Autorización Roles y permisos Acceso granular
Validación Bean Validation Datos seguros
HTTPS Certificados SSL Comunicación encriptada
Rate Limiting Spring Boot Actuator Prevenir abuso

14.3 Configuración de Seguridad Básica

// Ejemplo de configuración básica de seguridad
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Solo para desarrollo
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/**").authenticated()
                .anyRequest().permitAll()
            )
            .httpBasic(Customizer.withDefaults());
        
        return http.build();
    }
}

Este proyecto está configurado para desarrollo. Para producción se requieren cambios significativos.

  1. Base de Datos: Migrar de SQLite a PostgreSQL/MySQL

  6. CI/CD: Configurar pipeline de deployment
  7. Backup: Estrategia de respaldo de datos
  8. Escalabilidad: Configurar load balancer y múltiples instancias
